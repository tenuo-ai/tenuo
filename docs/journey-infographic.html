Tenuo User Journey

From setup to enforcement: how to integrate Tenuo into your AI agent infrastructure

Setup

1. Deploy Control Plane

Initialize the Control Plane service. It generates a root keypair (or loads from environment) and becomes the trust anchor. The public key is shared with all data planes for verification.

# Start Control Plane with environment variables # Generate enrollment token (shared secret for agent enrollment) $ export TENUO_ENROLLMENT_TOKEN=$(openssl rand -hex 32) # Option 1: Generate NEW keypair (dev/testing) # Don't set TENUO_SECRET_KEY - control plane calls Keypair::generate() # WARNING: Ephemeral key - not for production! # Option 2: Use EXISTING keypair (production) # Set TENUO_SECRET_KEY to hex-encoded secret key (64 hex chars = 32 bytes) # Control plane uses Keypair::from_bytes() to recreate keypair # Public key is derived from secret key via Ed25519 $ export TENUO_SECRET_KEY=your_existing_secret_key_hex_here # Optional: Bind address (default: 0.0.0.0:8080) $ export TENUO_BIND_ADDR=0.0.0.0:8080 $ tenuo-control # Output: Control Plane public key (derived from secret key) Public Key: f32e74b5b856... # Share this public key with all data planes for verification

Configuration

2. Configure Gateway

Define extraction rules: how HTTP request fields (path, headers, body) map to warrant constraint names. This tells the gateway which values to extract and check against the warrant.

# gateway-config.yaml tools: manage_cluster: constraints: cluster: from: path path: "cluster" # From /api/v1/clusters/{cluster}/scale required: true cost: from: body path: "metadata.estimatedCost" type: float max_replicas: from: body path: "spec.replicas" type: integer routes: - pattern: "/api/v1/clusters/{cluster}/scale" method: ["POST", "PUT"] tool: "manage_cluster"
$ tenuo validate-config --config gateway.yaml
Enrollment

3. Enroll Agents

Orchestrator generates a keypair locally, creates a Proof-of-Possession (PoP) signature, and enrolls with the Control Plane. The Control Plane issues a root warrant with the requested constraints (cluster, cost, etc.).

# Orchestrator generates its own keypair let keypair = Keypair::generate(); # Create PoP signature (proves ownership of keypair) let pop_msg = format!("enroll:{}:{}", pubkey_hex, timestamp); let pop_sig = keypair.sign(pop_msg.as_bytes()); # Enroll with Control Plane POST /v1/enroll { "public_key_hex": "a1b2c3...", "pop_signature_hex": "d4e5f6...", "enrollment_token": "...", "constraints": { "cluster": "staging-*", // Pattern: matches staging-web, staging-db, etc. "cost": "*", // Wildcard: any cost allowed initially "max_replicas": "*" // Wildcard: any replica count allowed initially } } # Response: Root warrant with constraints { "warrant_base64": "..." }

Delegation

4. Attenuate & Delegate

Orchestrator creates a narrower warrant for the worker by attenuating constraints. The child warrant can only restrict (shrink) the parent's scope‚Äînever expand it. This is cryptographically enforced.

# Orchestrator narrows constraints # Parent: cluster="staging-*", cost="*", max_replicas="*" # Child: cluster="staging-web", cost‚â§100.0, max_replicas‚â§10 let child = parent.attenuate() .constraint("cluster", Exact::new("staging-web")) .constraint("cost", Range::new(0.0, 100.0)) .constraint("max_replicas", Range::new(0, 10)) .authorized_holder(worker_keypair.public_key()) .build(&orchestrator_keypair)?;
Verification

5. Extract & Authorize

Gateway uses the config to extract values from the HTTP request, verifies the warrant chain signatures, checks PoP signature, and validates that extracted values satisfy all constraints. All verification happens offline (~25Œºs).

# Worker sends request with warrant chain POST /api/v1/clusters/staging-web/scale X-Tenuo-Chain: <base64 warrant chain> // Root ‚Üí Orchestrator ‚Üí Worker X-Tenuo-PoP: <signature> // Worker's Proof-of-Possession { "spec": { "replicas": 5 }, "metadata": { "estimatedCost": 50.0 }} # Gateway extracts using config (Phase 2) cluster: "staging-web" // from path /api/v1/clusters/{cluster}/scale cost: 50.0 // from body.metadata.estimatedCost max_replicas: 5 // from body.spec.replicas # Gateway verifies: ‚úì Chain signatures valid ‚úì PoP signature valid ‚úì cluster="staging-web" matches Exact("staging-web") ‚úì cost=50.0 within Range(0.0, 100.0) ‚úì max_replicas=5 within Range(0, 10) ‚Üí Authorized
